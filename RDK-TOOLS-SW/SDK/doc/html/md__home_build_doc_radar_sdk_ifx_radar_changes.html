Infineon Radar SDK for evaluation and development of applications with Infineon XENSIV 60GHz radar sensors.
Changelog Radar SDK

Radar SDK 3.2

Main new Features

  • The radar segmentation algorithm has been replaced by an updated and improved version: Segmentation Seamless Tracking.
  • Fetching of time-domain data from the radar sensors have been simplified. For more information see API Changes, Python and Matlab wrapper.
  • KissFFT has been replaced by muFFT, which is faster due to utilization of SIMD instructions. KissFFT dependencies are completely removed.
  • Downchirp configuration is now supported. Device configuration changed and uses now start_frequency instead of lower_frequency and end_frequency instead of upper_frequency.
  • Frame abstraction is replaced with Cube (and Matrix for CW) structures.
  • A lot of functions and structures names has been changed, mainly from device to avian and from radar to sensor. improving meaning and highlighting purpose of the functions and structures.
  • Some functions are marked deprecated and will be removed in 3.3.
  • Radar Baseboard MCU7 firmware: The UUID of the Radar Baseboard MCU7 is no longer saved in the EEPROM on the RF shield but directly on the board itself. As a consequence the UUID will change after updating to the latest Radar Baseboard MCU7 firmware. More information can be found in the Release Notes of the Radar Development Kit 3.2 release.

Bugfixes

  • Sometimes a wrong temperature value of about 140 degrees Celsius was read from BGT60ATR24C radar sensors. This problem has been fixed.
  • Following problems in constant wave (CW) mode with BGT60UTR11 have been fixed:
    • Calling ifx_avian_cw_measure_temperature could have take a long time.
    • Capturing a frame using ifx_avian_cw_capture_frame after reading TX power was resulting in a broken frame.
  • ifx_avian_cw_measure_temperature responsiveness has been fixed.
  • The function ifx_avian_metrics_get_limits erroneously returned 1 for the minimum TX power level. This has been fixed, the function now outputs the correct value tx_power_level.min=0.
  • The function ifx_avian_config_get_limits returned the wrong value for start_frequency_Hz and end_frequency_Hz. Both values were off by a factor of 1000.
  • Fixed a problem when calling ifx_avian_get_list and opening a radar device simultaneously from different threads. In this case either the returned list of available radar devices was empty or opening the radar device resulted in the error IFX_ERROR_NO_DEVICE.

  • The radar segmentation algorithm has been replaced by a new and better working version. In this case the API was also modified:
    • The members of the configuration structure ifx_Segmentation_Config_t have been changed. The number of configuration options has been reduced.

The function ifx_device_cw_capture_frame now sets the frame (ifx_Matrix_R_t) as a return value with rows = num_antennas, columns = num_samples. The API ifx_avian_cw_capture_frame creates a Constant Wave frame when called with a NULL option instead of the frame input parameter.

Python Wrapper

  • The fetching of frames has been simplified. It is no longer necessary to create a frame object and pass it to get_next_frame. The method create_frame_from_device_handle has been removed and the method get_next_frame now directly returns a numpy array with dimensions num_virtual_rx_antennas x num_chirps_per_frame x num_samples_per_frame. Here is an example how to fetch frames using the Python wrapper
    with Device() as device:
    # fetch 10 frames
    for frame_number in range(10):
    # frame has dimension num_rx_antennas x num_chirps_per_frame x num_samples_per_chirp
    frame = device.get_next_frame()
    # get matrix of dimension num_chirps_per_frame x num_samples_per_chirp for RX1
    matrix_rx1 = frame[0,:,:]
  • Renamed following functions:
    • get_shield_uuid to get_board_uuid
    • Methods get_config and set_config: lower_frequency_Hz and upper_frequency_Hz have been renamed to start_frequency_Hz and end_frequency_Hz, respectively. If start_frequency_Hz is lower than end_frequency_Hz an up-chirp is configured. Conversely, if start_frequency_Hz is larger than end_frequency_Hz a down-chirp is configured.
    • The optional argument of get_list is now expected to be an element of RadarSensor.
    • DeviceListEntry class field radar_device_type has been renamed to sensor_type.
  • Following structures have been renamed:
    • DeviceInfoStruct to SensorInfoStruct
  • The frame structure for Constant Wave mode has been changed to numpy matrix. The function capture_frame now returns a 2D numpy array with rows = num_antennas, columns = num_samples.
  • Downchirp configuration is now supported. Field lower_frequency_Hz has been changed to start_frequency_Hz and upper_frequency_Hz to end_frequency_Hz in DeviceConfigStruct class.
  • Field hp_cutoff in DeviceBasebandConfigStruct is no longer an enum value, but a uint16_t value representing actual high pass filter cutoff frequency in Hz.
  • Field aaf_cutoff in DeviceBasebandConfigStruct is no longer an enum value, but a uint16_t value representing actual AAF cutoff frequency in Hz.
  • Added enum BGT120UTR24 to RadarSensor class.
  • Renamed translate_metrics_to_config to metrics_to_config.
  • Added function metrics_from_config.
  • Added example range-angle-map.py utilizing updated doppler module and Digital Beam Forming.

Radar SDK 3.1

Bugfixes

  • Fixed a bug in ifx_device_set_config which caused that the chirp-to-chirp-time and the frame time were not set correctly in some cases.
  • Fixed a bug in functions ifx_vec_set_range_r and ifx_vec_set_range_c. The behavior has changed and now they fill the vector from offset value place for a range value of places, meaning they fill the vector from offset to offset+range.
  • Fix the bug that the functions ifx_mat_set_row_r and ifx_mat_set_row_c set an error if the number of columns of the matrix equals count.
  • Missing checks of parameters for ifx_rdm_run_rc and ifx_rdm_run_r have been added.
  • A check has been added to avoid a crash when connecting to a radar shield with a broken EEPROM.

New Features

  • The radar sensor BGT60UTR11 now works out of the box with the presence sensing application.
  • Support for the Avian radar sensor BGT60UTR13D has been added.
  • The new tool bgt60-configurator-cli has been added which allows to convert a json configuration to register values.

API Changes

Changes:

  • The type of the second parameter of ifx_ppfft_set_mean_removal_flag has been changed from uint8_t to bool.
  • The declaration of the function ifx_presence_sensing_get_config_defaults has changed. There is a new input parameter of type ifx_Device_Radar_Type_t which specifies the device type for which the configuration should be given. The parameter of type ifx_Device_Config_t is now an output parameter. The implementation of the function now gives in this output parameter the default device configuration for presence sensing.
  • The declaration of the function ifx_segmentation_config_get_defaults has changed. There is a new input parameter of type ifx_Device_Radar_Type_t which specifies the device type for which the configuration should be given. The first parameter of type ifx_Device_Config_t is now an output parameter. The implementation of the function now gives in this output parameter the default device configuration for segmentation.
  • The members of the enum ifx_Device_Shield_Type_t have been renamed.
  • The functions ifx_vect_median_range_r and ifx_vect_median_r have been renamed to ifx_vec_median_range_r and ifx_vec_median_r, respectively.
  • The function for creating a Butterworth filter object has been renamed from ifx_signal_filter_butterwort_create_r to ifx_signal_filter_butterworth_create_r.
  • The structure ifx_Device_Info_t has a new member shield_type. The field is also available via the Python and Matlab wrappers.
  • The function ifx_signal_correlate_r has one more parameter to specify the mode (either full or same).
  • Avian Python wrapper: The method get_list now expects the shield type to be of type ShieldType instead of a string.

Functions added:

  • The function ifx_device_config_get_limits has been added.
  • The function ifx_device_get_device_type has been added.
  • The functions ifx_vec_dot_r and ifx_vec_dot2_r to compute the dot product between two vectors have been added.
  • The function ifx_device_cw_destroy has been added.
  • The functions ifx_mat_clone_r, ifx_mat_clone_c to clone matrices have been added.
  • The functions ifx_device_cw_get_sampling_rate_limits and ifx_device_cw_get_sampling_rate have been added to the constant wave (CW) mode.

Other Changes

  • Reading the temperature from an Avian radar sensor is now cached. When calling ifx_device_get_temperature twice within a short amount of time the function will return the same temperature value. This change prevents performance degradations and stability issues when fetching time-domain data from the radar sensor and reading the temperature very often.
  • The format of the return value of ifx_sdk_get_version_string has changed: It returns now a string with major number, minor number and patch level separated by dots.

Radar SDK 3.0

Bugfixes

This is an incomplete list of fixed bugs:

  • An error when measuring the TX power with ATR24C has been fixed.
  • Bugs in RangeDopplerMap and in RangeSpectrum have been fixed.
  • A bug has been fixed that caused errors if the radar shield was plugged into the backside of the RadarBaseboardMCU7

New Features

  • The Python and Matlab wrappers have been extended.
  • The documentation has been improved. The page Introduction to Radar SDK has been rewritten, extended and clarified. The pages about the Python wrapper and the Matlab wrapper have been extended with a tutorial illustrating how the wrappers can be used to implement a simple anti-peeking/presence-detection algorithm.
  • It is now possible to extract the register values of the radar sensor corresponding to the current configuration.
  • The DeviceControl layer has been refactored. It is now possible to open a mock device that does not require a physical sensor attached to the computer. That is particularily useful when converting a device configuration to a register list.
  • Support for the radar sensor BGT60UTR11 has been added. Some applications still need more than one RX antenna and might need slight modification when used with a UTR11. The CW mode is not fully supported with BGT60UTR11 radar sensors yet.

API Changes

  • The function ifx_cube_slice_abs_r has been removed.
  • Functions to calculate range, speed and frequency axis for Range Spectum and Range Doppler Map can be used from SpectrumAxis.h module.
  • The functions ifx_rs_calc_range_axis and ifx_rs_calc_freq_axis, have been removed from RangeSpectrum.h module.
  • The functions ifx_rdm_calc_range_axis and ifx_rdm_calc_speed_axis, have been removed from RangeDopplerMap.h module.
  • FFTW3 as a backend for FFT computations has been removed. KissFFT is now the only supported backend.
  • The function ifx_math_abs_c has been removed. One can use the function ifx_complex_abs instead.
  • The option for memory tracking has been removed. The function ifx_mem_get_total_alloc_size has been removed as well.
  • The functions ifx_vec_setrng_r and ifx_vec_setrng_c have been renamed to ifx_vec_set_range_r and ifx_vec_set_range_c, respectively
  • The function ifx_device_get_center_frequency has been renamed to ifx_devconf_get_center_frequency.
  • The function ifx_device_get_bandwidth has been renamed to ifx_devconf_get_bandwidth.
  • The function ifx_device_get_chirp_repetition_time has been renamed to ifx_devconf_get_chirp_repetition_time.
  • The function ifx_device_count_rx_antennas has been renamed to ifx_devconf_count_rx_antennas.
  • The enum ifx_Device_Type_t has been renamed to ifx_Device_Shield_Type_t.
